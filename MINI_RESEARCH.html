Figure 1. Landing Page

🖼️
Screenshot — Landing Page
http://localhost:5000/
Figure 1. The IronGate landing page at /

The first page users see when visiting the system at http://localhost:5000. Displays the IronGate brand, a hero headline, and a brief description of the system. Contains two call-to-action buttons — Enter Admin Panel (routes to the login form) and Explore Features (scrolls down the page).

The landing page is divided into five sections:

  • Hero — system name, tagline, and key stats (VGG-Face model, 0.40 cosine threshold, 50+ photos per member, 100% local — no cloud)
  • Features — nine feature cards covering face recognition, access control, emotion detection, member management, payments, analytics, gym floor monitoring, multi-admin roles, and backup/restore
  • How It Works — six numbered steps from face detection to entry logging
  • Tech Stack — cards for Python, Flask, DeepFace, TensorFlow, OpenCV, VGG-Face, NumPy/Pandas, Pillow, Watchdog, and JSON/CSV storage
  • Team & CTA — Group 9 members with roles and a final call-to-action button linking to the admin login
No credentials are entered here. The landing page is fully public and simply introduces the system, routing the user to the correct login form.

Figure 2. Admin Login Page

🖼️
Screenshot — Admin Login Page
http://localhost:5000/login
Figure 2. The admin login form at /login

Located at /login. The only entry point into the admin panel. Displays the IronGate logo, a username field, and a password field.

  • Successful login redirects to the Dashboard
  • Failed attempts increment a per-IP counter
  • After 5 failed attempts, the IP is locked out for 5 minutes — a message shows the remaining lockout time
  • Sessions expire automatically after 2 hours of inactivity
  • Passwords are hashed with SHA-256 — never stored in plaintext
  • A ← Back to Home link returns to the landing page

Figure 4. Members Page

🖼️
Screenshot — Members Page
http://localhost:5000/members
Figure 4. The members list at /members

Full list of all registered gym members. The search bar filters by name, membership type, status, contact number, or email in real time.

Table columns: row number, profile photo, full name, membership type (Monthly / Quarterly / Annual), status (Active / Expired / Suspended / Frozen), start date, expiry date, days left (red ≤7d, yellow ≤30d, green otherwise), total visits, contact number, and action buttons.

Action buttons per member: View profile, Edit details, 📷 Add more face photos, 💰 Payment history, ❄️ Freeze/Unfreeze, Suspend/Renew, and Remove (with confirmation). Membership status auto-updates on page load — expired dates are detected and status is set to Expired automatically.

Figure 5. Add Member & Face Capture

🖼️
Screenshot — Add Member Form
http://localhost:5000/members/add
Figure 5a. Add Member form at /members/add

A two-section form: Basic info (full name, membership type, status, start/expiry dates — expiry auto-calculates) and Personal info (gender, birthday, address, contact, email, emergency contact, notes). After saving, the system generates a folder name and redirects to Face Capture.

🖼️
Screenshot — Face Capture
http://localhost:5000/capture-stream
Figure 5b. Face capture progress page

The webcam opens and automatically captures 50 face photos when a face is detected. Each photo is cropped, resized to 224×224 px, and enhanced with CLAHE before saving to dataset/<FolderName>/. A progress bar shows the count. The page redirects to Members automatically when done.

Figure 7. Entry Log

🖼️
Screenshot — Entry Log
http://localhost:5000/entry-log
Figure 7. The entry log at /entry-log

Every face recognition event is recorded here automatically. A date filter narrows records to a specific day. Stats row shows Total, Granted, and Denied counts for the current filter.

Table columns: row number, member name, timestamp, access result (color-coded pill), deny reason, detected emotion, detected gender, estimated age, confidence score (%).

Export CSV downloads all visible records as a .csv file. Clear Log permanently deletes all entry records (with confirmation). Records are stored in entry_log.csv and appended automatically by the face recognition engine.

Figure 8. Gym Floor — Live Occupancy

🖼️
Screenshot — Gym Floor
http://localhost:5000/gym-floor
Figure 8. Live gym occupancy at /gym-floor

Shows who is currently inside the gym in real time. Members are automatically checked in when face recognition grants them access. Each card shows the member photo, full name, check-in time, and a Check Out button.

The occupancy counter updates every 10 seconds via the /api/gym-status endpoint. Clicking Check Out calls /api/checkout and removes the member from the floor view. The check-in state is in-memory and resets when the server restarts.

Figure 9. Payments

🖼️
Screenshot — Payments
http://localhost:5000/payments
Figure 9. Payment records at /payments

All recorded membership payments. Summary stats show total collected (₱) and total transaction count.

Table columns: row number, member name, amount (₱), payment method (Cash / GCash / Card / Bank Transfer — color-coded), description, date, recorded by (admin username).

Record Payment (/payments/add) — form fields: member selector, amount, payment method, payment date (defaults to today), description/notes. Member Payment History (/payments/member/<name>) shows all payments for a single member with a running total. All transactions are logged in the Audit Log.

Auto-generated insights from the entry log data. Summary stats (4 cards): Total Members, Granted (last 30 days), Denied (last 30 days), Peak Hour.

Figure 11. Backups

🖼️
Screenshot — Backups
http://localhost:5000/backups
Figure 11. Data backup and restore at /backups

Create Backup — one click saves a timestamped snapshot of all five data files: members.json, admins.json, payments.json, entry_log.csv, audit_log.csv. Stored in the backups/ folder.

Backup list — shows each backup's date and the files it contains. Restore — clicking Restore overwrites the current live data files with the backup copies. A confirmation prompt prevents accidental restores. All backup and restore actions are recorded in the Audit Log.

Figure 12. Audit Log

🖼️
Screenshot — Audit Log
http://localhost:5000/audit-log
Figure 12. Admin audit trail at /audit-log (Superadmin only)

A complete, tamper-evident trail of every admin action. Table columns: row number, timestamp, admin username, action type (color-coded pill), target, details.

Action Colors

  • ■ member_added — green
  • ■ member_deleted — red
  • ■ payment_recorded — yellow
  • ■ freeze/activate — blue
  • ■ others — grey

Security Notes

  • Superadmin-only access
  • Append-only from the UI
  • Cannot be cleared via interface
  • Thread-safe writes (locked)
  • Stored in audit_log.csv

Figure 13. Admin Accounts

🖼️
Screenshot — Admin Accounts
http://localhost:5000/register
Figure 13. Admin account management at /register (Superadmin only)

Left panel — Create New Admin Account: full name, username (min 3 chars), password (min 6 chars), confirm password, role selector (Staff or Superadmin). Duplicate usernames are rejected. Passwords are SHA-256 hashed before storage.

Right panel — Existing Accounts: lists all admin accounts with name, role pill, username, and creation date. A Remove button deletes any account except the currently signed-in admin's own account.

Superadmin

  • Full access to all pages
  • Audit Log visibility
  • Admin account management
  • All member/payment operations

Staff

  • Manage members
  • View entry log & analytics
  • Record payments
  • Create/restore backups
  • Cannot access Audit Log or Admin Accounts
